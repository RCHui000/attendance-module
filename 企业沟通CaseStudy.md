# 项目审批与工时系统 Case Study

面向企业老板、管理层和业务负责人沟通展示版

## 一句话概括

我们为一家以项目制交付为核心的企业，搭建了一套“工时填报 + 项目审批 + 部门汇总 + 成本分析”的内部管理系统，把原本分散在 Excel、微信群、人工汇总里的项目人力投入，沉淀为可审批、可追溯、可分析的数据资产。

这不是单纯做一个考勤表，而是把“员工每天做了什么项目、项目负责人是否认可、部门负责人是否确认、最终形成多少项目人工成本”这条管理链路跑通。

## 项目背景

很多设计、咨询、工程、项目管理类企业都会遇到同一个问题：

- 员工每天做了很多项目，但项目投入靠事后回忆或人工统计。
- 项目经理知道项目忙不忙，但老板很难实时看到哪个项目真正消耗了多少人。
- 部门负责人要管人、管产值、管成本，却缺少统一的数据入口。
- 财务或经营分析依赖 Excel 汇总，数据口径容易不一致。
- 项目成本、人员效率、部门贡献无法形成长期可复用的管理数据。

本项目的核心目标，是把这些“靠人盯、靠表拼、靠会议问”的管理动作，固化成一套轻量但完整的业务系统。

## 客户痛点

### 1. 项目人力成本看不清

企业并不缺项目合同额、回款数据，真正难的是看清每个项目消耗了多少内部人力。没有稳定的工时数据，项目利润往往只能粗算，难以判断哪些项目看似赚钱、实际吃掉了大量人力。

### 2. 审批链条不清晰

员工填报后，谁来确认项目工时是否合理？项目负责人确认之后，部门负责人是否还需要从部门管理角度复核？如果审批只停留在口头或群消息里，后续追溯很困难。

### 3. 数据分散，重复劳动多

员工、项目、组织、合同、薪酬、周表、加班、审批记录分散在不同文件和沟通场景里。管理人员需要反复整理、核对、复制粘贴，越到月底越容易出错。

### 4. 老板缺少经营视角看板

老板关心的不是每一张周表本身，而是：

- 本月哪些项目最耗人？
- 哪些部门投入最多？
- 哪些员工在参与关键项目？
- 项目人工成本和回款、合同额之间是什么关系？
- 审批通过的数据是否可信？

传统 Excel 可以做结果统计，但很难保证前端采集、审批确认、后端分析是一条闭环。

## 解决方案

我们设计并落地了一套项目审批与工时系统，围绕企业项目管理的真实流程展开：

```text
员工填周表
-> 项目负责人按项目块审批
-> 部门负责人汇总审批
-> 审批通过后进入 BI 统计
-> 老板和管理层查看项目、部门、人员三类分析
```

系统覆盖五个核心模块：

| 模块 | 解决的问题 |
| --- | --- |
| 我的周表 | 员工按周填报项目工日和加班，支持保存草稿、提交审批、退回修改 |
| 审批中心 | 项目负责人审批项目工时，部门负责人最终确认整张周表 |
| 数据看板 | 汇总活跃项目、投入工日、人力成本、项目排行等经营指标 |
| 项目列表 | 管理项目基础信息、负责人、投入工日、人工成本、回款和毛利口径 |
| 员工与组织 | 维护员工、部门、岗位、角色、合同、薪酬等基础数据 |

## 关键设计

### 1. 以“周表”为业务入口

员工不需要每天打开复杂系统，只需要按周选择项目、填写每天投入工日。系统控制每天总工日不超过 1，避免基础数据失真。

### 2. 以“项目块”为审批单元

一张周表里可能包含多个项目。系统不会简单地把整张表一次性丢给一个人，而是按项目拆分给对应项目负责人审批。这样项目经理只确认自己负责的项目，职责清晰。

### 3. 以“部门汇总”为最终口径

项目负责人确认项目工时后，还需要部门负责人从人员管理和部门成本角度做最终确认。只有最终通过的周表，才进入正式 BI 和成本统计。

### 4. 权限按角色和组织边界控制

系统区分员工、项目负责人、部门负责人、管理员等角色。普通员工只处理自己的周表；部门负责人可查看和维护本部门范围内数据；管理员拥有全局维护能力。

### 5. 统计只认“审批通过”的数据

草稿、审批中、退回状态不会进入正式成本分析。这样老板在看 BI 时，看到的是经过业务负责人确认的数据，而不是未经校验的临时填报。

## 管理价值

### 对老板

系统把项目人力投入从“感觉”变成“数据”。老板可以更快判断项目是否健康、部门是否饱和、人员投入是否匹配产出。

### 对项目负责人

项目负责人只需要确认自己负责项目的工时，不再从整张 Excel 里人工筛选。审批动作有记录，后续争议可追溯。

### 对部门负责人

部门负责人能看到本部门员工本周做了哪些项目、投入多少工日，并在最终审批时把关部门资源使用情况。

### 对员工

员工按周填写，流程简单；被退回后可以按意见修改；不用反复在多个表格或群消息里汇报同一类信息。

### 对财务和经营分析

已审批周表可以沉淀为项目成本和人员效率分析的基础数据，后续可以继续接入合同、回款、利润、奖金核算等更深层经营场景。

## 技术落地方式

本项目采用前后端分离和数据库权限控制结合的方式：

- 前端：React + TypeScript + Vite。
- UI：Tailwind CSS、shadcn/ui 风格组件、Lucide 图标。
- 图表：Recharts。
- 数据层：PostgreSQL + Supabase 风格运行态。
- 认证：GoTrue 登录认证。
- 接口：PostgREST + 业务 RPC。
- 实时刷新：Supabase Realtime + 浏览器多标签同步兜底。
- 部署：支持 NAS 局域网部署，也支持阿里云 ECS 云端部署。

对老板沟通时，可以把这部分压缩成一句话：

> 系统不是临时脚本，而是按企业内部系统标准做了账号、权限、审批、数据库、部署和运维文档，可以在局域网和云服务器上运行。

## 项目过程亮点

### 1. 从真实业务流程出发，而不是先堆功能

项目一开始就围绕“工时如何变成可信经营数据”来设计，而不是只做填表页面。因此审批、权限、统计口径和部署都被纳入同一套方案。

### 2. 逐步迭代，边用边校准

系统已经经历多个版本迭代，从基础周表到项目块审批、部门汇总、BI 看板、权限补强、云端预部署，不是一次性纸面方案。

### 3. 把复杂度留在系统里，把操作留得简单

审批路由、权限策略、统计口径、数据聚合在系统内部处理。用户看到的是“填报、审批、查看”这些直接动作。

### 4. 对未来扩展做了预留

当前版本已经识别出下一阶段可优化方向，例如：

- 同一项目按不同部门配置不同项目负责人。
- 项目块级审批状态显式化。
- 员工资料保存事务化。
- BI 汇总数据仓库化。
- NAS 与云端数据同步治理。

这说明系统不是一次性工具，而是可以持续演进的管理平台。

## 对外沟通建议

### 30 秒版本

我们做过一个项目制企业的内部管理系统，把员工周报、项目负责人审批、部门负责人确认和老板 BI 看板串起来。以前项目人工成本靠 Excel 和人工汇总，现在员工按周填报，负责人在线审批，最终通过的数据自动进入项目、部门、人员三类分析。它解决的不是单纯考勤，而是项目人力成本和经营管理透明化。

### 3 分钟版本

这个项目来自一个典型项目制企业的管理需求。老板想知道每个项目到底用了多少人、花了多少人工成本，但原来数据分散在 Excel、聊天记录和人工统计里，口径不统一，也不好追溯。

我们搭了一套项目审批与工时系统。员工每周填写自己参与的项目和每天工日；提交后，系统按项目拆给项目负责人审批；项目都通过后，再交给部门负责人做最终汇总确认。只有最终通过的数据，才进入 BI 看板。

老板可以从项目、部门、人员三个视角看数据：哪些项目投入最多、哪个部门参与了哪些项目、每个人本月主要做了什么。项目负责人和部门负责人也有自己的审批中心和已审核记录。

这个项目的价值在于，它把工时填报从一个行政动作，变成了项目成本核算和经营分析的数据入口。后续还可以继续扩展到回款、毛利、奖金、产能预测等场景。

## 老板常问问题与回答

### 这个是不是只适合考勤？

不是。普通考勤回答“人有没有上班”，这套系统回答“人把时间投到了哪个项目、由谁确认、形成了多少项目成本”。

### 员工会不会觉得麻烦？

系统按周填报，不要求每天复杂操作。员工只需要选择项目、填写每天工日、保存或提交。真正复杂的是审批路由和统计口径，系统在后台处理。

### 数据准不准？

系统通过三层机制提高可信度：员工填报、项目负责人确认项目工时、部门负责人最终确认。并且只有最终审批通过的数据才进入正式统计。

### 能不能适配我们公司的组织架构？

可以。系统已经支持员工、部门、岗位、负责人、角色权限等基础数据维护。不同企业可以按自己的部门和项目结构配置。

### 能不能先小范围试点？

可以。建议先选一个部门或一类项目试点，跑通员工填报、项目审批、部门确认和看板分析，再逐步扩展到全公司。

### 能不能部署在内网？

可以。当前项目已经支持 NAS 局域网部署，也预部署到阿里云 ECS。企业可以根据安全要求选择内网、云端或混合方式。

## 后续商业化包装方向

后续如果要把这个案例包装成对外方案，可以命名为：

- 项目型企业工时与成本管理系统。
- 项目人力成本核算平台。
- 工时审批与项目经营分析系统。
- 专业服务企业项目成本 BI 系统。

建议对外主张不要叫“考勤系统”，而要叫“项目人力成本管理系统”。“考勤”容易被理解成打卡，“项目人力成本”才是老板愿意付费解决的问题。

## 结论

这个项目的核心成果，是把项目制企业最难沉淀的一类数据，即“人力投入”，从零散、滞后、不可追溯的人工统计，变成了一条在线流程：

```text
填报有入口
审批有责任
数据有口径
成本可分析
部署可落地
未来可扩展
```

这份经验可以作为后续和其他企业老板沟通的典型案例：先讲他们熟悉的痛点，再展示我们已经落地过的闭环，最后引导到“可以从一个部门试点，逐步变成企业经营数据底座”。
